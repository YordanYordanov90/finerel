# AI Workflow Rules

## Approach

Build FinRel incrementally using a spec-driven workflow. The seven context files
in this folder define what to build, how to build it, and the current state of
progress. Always implement against these specs — do not infer or invent behavior
not defined here. When a decision is missing, add it as an open question in
`progress-tracker.md` and resolve it before writing code.

## Scoping Rules

- Work on one feature unit at a time.
- Prefer small, verifiable increments over large speculative changes.
- Do not combine unrelated system boundaries in a single implementation step.
- The cron agent logic (`lib/agent/`, `app/api/cron/`) and the dashboard UI are separate concerns — implement and verify them separately.

## When to Split Work

Split an implementation step if it touches:

- Both the cron agent logic and the dashboard UI in the same step.
- A schema change and a UI change at the same time.
- More than one tool in the agent (each tool is its own unit).
- Any behavior not clearly defined in the context files.

If a change cannot be verified end to end quickly, the scope is too broad — split it.

## Handling Missing Requirements

- Do not invent product behavior not defined in the context files.
- If a requirement is ambiguous, resolve it in the relevant context file before implementing.
- If a requirement is missing, add it as an open question in `progress-tracker.md` before continuing.
- The `financial-researcher-agent.md` spec is the root document — any conflict between it and these context files should be resolved in favor of the spec, then the context file updated.

## Protected Files

Do not modify the following unless explicitly instructed:

- `components/ui/*` — shadcn-generated UI components. Use the CLI to add or update.
- `lib/db/migrations/*` — generated by Drizzle Kit. Never edit migration files by hand.
- Any third-party library internals.

## Keeping Docs in Sync

Update the relevant context file whenever implementation changes:

- Stack or deployment decisions → `architecture.md`
- New coding conventions or security rules → `code-standards.md`
- UI token changes or new layout patterns → `ui-context.md`
- Feature scope changes → `project-overview.md`
- Completed work, open questions, decisions → `progress-tracker.md`

## Security Checkpoints

Before completing any implementation unit, verify:

- All external input validated with Zod at the boundary.
- All database queries include `userId` filter.
- No secrets, API keys, or tokens in client components, logs, or API responses.
- QStash signature verification in place on every cron endpoint.
- Demo mode enforced server-side — no client-sent flags trusted.

## Before Moving to the Next Unit

1. The current unit works end to end within its defined scope.
2. No invariant defined in `architecture.md` was violated.
3. `progress-tracker.md` reflects the completed work and any decisions made.
4. `npm run build` passes with zero errors.
5. The security checkpoints above have been reviewed for this unit.
